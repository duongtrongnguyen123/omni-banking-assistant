## Critical issues (block demo)

### Critical-1: `/api/session/reset` raises `AttributeError: can't set attribute`

`backend/app/routes/chat.py:129` does:

```python
s = session_for(user_id)
s.current_draft = None
s.current_contact_draft = None
s.current_schedule_draft = None
s.history.clear()
```

but in `backend/app/context/session.py` `current_draft`,
`current_contact_draft`, and `current_schedule_draft` are now **read-
only `@property`** wrappers over a `SessionBackend` (Redis / in-memory).
Setting them raises `AttributeError`. The Redis-sessions refactor
intentionally removed the public setter but the chat route never moved
to `clear_draft()` / `clear_contact_draft()` / `clear_schedule_draft()`.

Repro:

```
$ curl -s -X POST http://localhost:8000/api/session/reset -H 'x-user-id: u_an'
Internal Server Error
```

Server log:

```
File "backend/app/routes/chat.py", line 129, in reset_session
    s.current_draft = None
AttributeError: can't set attribute
```

This is **not caught by `make verify`** because no test or KB script
hits the endpoint:

```
$ grep -rln 'session/reset\|reset_session' backend/tests/ backend/scripts/
(no matches)
```

The "fresh chat" button in the UI silently 500s in the offline demo
recorder path and during multi-scenario E2E runs.

**Fix** (4 lines):

```python
@router.post("/session/reset")
def reset_session(user_id: str = Depends(current_user)) -> dict:
    s = session_for(user_id)
    s.clear_draft()
    s.clear_contact_draft()
    s.clear_schedule_draft()
    s.history.clear()
    return {"ok": True, "user_id": user_id}
```

Add a 5-line test (`tests/test_session_reset.py`) hitting the endpoint
both with and without an in-flight draft, so this can't regress again.

## Suggested next 3 features

Optimising for **judge-visible delta** in a 5-minute pitch, given
everything that's already shipped:

1. **Audit-explain UI** — every flagged transfer or budget overshoot
   already has a structured reason from the rule engine. Add a chat
   card that, when the user taps "Vì sao?" on a warning, surfaces the
   decision tree (rule fired, prior fact, threshold). Closes the loop
   on "An toàn vẫn được đảm bảo" with a teachable artefact judges can
   replay in the demo. Branch `feat/audit-explain` exists (per stale
   `branch-status.md`) and the prerequisite `Store.audit_of` is small
   to add — confirm before merging.

2. **Insights → chat action linkage** — `/api/insights/summary` already
   returns MoM rows like `{food: this:60k, last:120k, delta_pct:-50}`.
   Add an "explain" CTA on each row that injects a chat message ("vì
   sao ăn uống giảm 50%?") which runs a history aggregate filtered to
   that category. Same architecture suggestion as the prior audit — it
   still hasn't shipped, and it's the cheapest way to make the
   insights card feel interactive instead of static.

3. **HTTP smoke pass over all 56 routes** — Critical-1 is a class of
   bug: route handlers that never get touched by `make verify`. A 56-
   line pytest that hits every route registered on `app.routes` with a
   smoke payload, asserting `< 500` and Pydantic-shaped response, would
   have caught the `current_draft` setter regression in seconds. Not a
   user-visible feature, but a force-multiplier on every future merge
   wave.
